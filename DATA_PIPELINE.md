# DATA_PIPELINE.md

## Цель

Сделать процесс наполнения Relogator управляемым: агенты собирают данные не в код компонентов и не в свободный текст, а в структурированные сущности, которые потом можно проверить, обновить, перенести в Supabase и использовать в правилах подбора.

Главный принцип:

`источник -> факт -> условие -> этап -> переход -> правило -> маршрутный план -> персональная оценка`

ИИ может помогать собирать, нормализовать и проверять данные, но основной пользовательский путь должен работать без обязательного ИИ.

## Почему не одно дерево

Дерево подходит для анкеты: какой вопрос задать следующим.

Для стран, маршрутов и документов дерево недостаточно. Одно условие может влиять сразу на много маршрутов:

- загранпаспорт влияет почти на все страны;
- подтверждаемый доход влияет на визы удаленной работы, ВНЖ по доходу, аренду и банки;
- дети добавляют документы в разных странах;
- справка о несудимости нужна не везде, но влияет на многие долгие статусы;
- срок выезда меняет доступность маршрутов.

Поэтому базовая модель должна быть графом правил, фактов и состояний во времени:

- страны;
- источники;
- маршруты;
- этапы пребывания;
- переходы между этапами;
- законные действия;
- дедлайны;
- запасные планы;
- требования;
- документы;
- шаги;
- условия применимости;
- правила оценки;
- связи между ответами анкеты и маршрутами.

## Целевая схема данных

Продукт читает проверенный каталог из Supabase Postgres:

- `content_countries` - страны и статус глубины проверки;
- `content_sources` - официальные и редакционные источники;
- `content_routes` - маршруты, требования, шаги, риски и `decision_graph`.

Структурированные файлы в `data/` остаются резервом, сидами и удобным форматом для работы агентов. После проверки изменений агент запускает `npm run validate:data`, затем инженер или агент с серверным ключом запускает `npm run sync:content-db`, чтобы загрузить текущий каталог в Supabase.

Резервная структура файлов:

```text
data/
  countries/
    GE.json
    RS.json
  sources/
    GE.json
    RS.json
  routes/
    GE/
      visa-free-one-year.json
    RS/
      temporary-residence-work-business.json
  requirements/
    passport.json
    income.json
    translations.json
  documents/
    criminal-record-certificate.json
    birth-certificate.json
  rules/
    route-availability.json
    dimension-scoring.json
  route-graph/
    stage-nodes.json
    transition-edges.json
    legal-actions.json
    deadlines.json
    fallback-plans.json
  drafts/
    agent-submissions/
```

Публичный продукт должен читать только проверенные данные. Черновики агентов должны лежать отдельно.

## Основные сущности

### Source

```ts
type Source = {
  id: string
  title: string
  url: string
  sourceType:
    | "official_body"
    | "law"
    | "consulate"
    | "statistics"
    | "partner"
    | "editorial_note"
  countryCode?: string
  language: string
  lastCheckedAt: string
  appliesToCitizenship: "RU"
  confidence: "high" | "medium" | "low"
  notes?: string
}
```

### Country

```ts
type Country = {
  code: string
  nameRu: string
  nameEn: string
  region: string
  priority: 1 | 2 | 3
  publicationStatus:
    | "reviewed"
    | "needs_review"
    | "stale"
    | "reference_only"
    | "archived"
  sourceIds: string[]
  lastReviewedAt: string
}
```

### Route

```ts
type Route = {
  id: string
  countryCode: string
  title: string
  goalTags: string[]
  entryType:
    | "visa_free"
    | "e_visa"
    | "visitor_visa"
    | "temporary_residence"
    | "residence_permit"
    | "work"
    | "business"
    | "family"
    | "study"
  publicationStatus:
    | "draft"
    | "reviewed"
    | "partner_reviewed"
    | "stale"
    | "archived"
  sourceIds: string[]
  requirementIds: string[]
  documentIds: string[]
  stepIds: string[]
  lastReviewedAt: string
  confidence: "high" | "medium" | "low"
}
```

### StageNode

```ts
type StageNode = {
  id: string
  routeId: string
  countryCode: string
  type:
    | "prepare"
    | "entry"
    | "registration"
    | "short_stay"
    | "extension"
    | "status_application"
    | "waiting_decision"
    | "maintain_status"
    | "exit"
  title: string
  description: string
  sourceIds: string[]
  lastReviewedAt: string
  publicationStatus: "reviewed" | "needs_review" | "reference_only" | "stale"
}
```

### TransitionEdge

```ts
type TransitionEdge = {
  id: string
  routeId: string
  fromStageId: string
  toStageId: string
  transitionType:
    | "extend"
    | "apply_status"
    | "change_basis"
    | "exit_and_reenter"
    | "switch_to_work"
    | "switch_to_study"
    | "switch_to_family"
    | "finish_route"
  legalActionIds: string[]
  conditionIds: string[]
  deadlineIds: string[]
  fallbackPlanIds: string[]
  sourceIds: string[]
  confidence: "high" | "medium" | "low"
}
```

### FallbackPlan

```ts
type FallbackPlan = {
  id: string
  title: string
  appliesWhen: RuleCondition[]
  actions: string[]
  risks: string[]
  sourceIds: string[]
  needsSpecialistReview: boolean
}
```

### Requirement

```ts
type Requirement = {
  id: string
  title: string
  appliesWhen: RuleCondition[]
  hardBlockerWhen?: RuleCondition[]
  affects: ("documents" | "cost" | "approvalRisk" | "speed" | "adaptation")[]
  sourceIds: string[]
  lastReviewedAt: string
}
```

### RuleCondition

```ts
type RuleCondition = {
  field: string
  operator: "equals" | "not_equals" | "in" | "not_in" | "gte" | "lte" | "exists"
  value: unknown
}
```

### RouteEvaluation

```ts
type RouteEvaluation = {
  routeId: string
  availability: "available" | "conditional" | "blocked" | "unknown"
  dimensions: {
    documents: 1 | 2 | 3 | 4 | 5
    cost: 1 | 2 | 3 | 4 | 5
    approvalRisk: 1 | 2 | 3 | 4 | 5
    speed: 1 | 2 | 3 | 4 | 5
    adaptation: 1 | 2 | 3 | 4 | 5
  }
  blockers: string[]
  unlocks: string[]
  sourceIds: string[]
  lastReviewedAt: string
}
```

## Как будет управляться работа

Не запускать десять агентов без системы. Работать волнами.

Одна волна = 3-5 агентов, один ответственный интегратор, один проверяющий.

### Роли в команде

1. Главный постановщик

Роль: ты или текущий главный Codex.

Задачи:

- выбрать цель волны;
- определить список стран или маршрутов;
- выдать промты агентам;
- принять или отклонить результат;
- решить, что идет в продукт, а что остается черновиком.

2. Агент-сборщик данных

Количество: 2-3 агента на волну.

Задачи:

- собрать официальные источники;
- извлечь факты;
- не делать выводов без источника;
- заполнить черновые структурированные записи.

Не имеет права:

- менять код интерфейса;
- менять правила оценки;
- публиковать данные как проверенные;
- писать юридические обещания.

3. Агент-проверяющий

Количество: 1 агент на волну.

Задачи:

- проверить источники сборщиков;
- найти противоречия;
- отметить устаревшие или сомнительные данные;
- присвоить `confidence`;
- составить список ручных вопросов к специалисту.

4. Агент-моделировщик правил

Количество: 1 агент, не всегда нужен.

Задачи:

- превратить факты в условия и правила;
- предложить `Requirement`, `RuleCondition`, `RouteEvaluation`;
- не добавлять страновые факты без источника.

5. Агент-разработчик

Количество: 1 агент на реализацию.

Задачи:

- подготовить структуру данных;
- подключить чтение структурированных файлов;
- реализовать оценку маршрутов;
- обновить интерфейс;
- добавить тесты.

Не должен:

- сам придумывать миграционные правила;
- хардкодить страны и маршруты в компоненты;
- смешивать справочные данные с React-компонентами.

6. Агент-дизайнер

Количество: 1 агент, когда есть готовая логика или понятный экран.

Задачи:

- улучшить экран анкеты, результатов, маршрута;
- использовать дизайн-ограничения из `AGENTS.md`;
- проверять мобильный вид;
- не превращать продукт в рекламный лендинг.

4. После сборщиков запустить проверяющего источников: он сверяет официальность источников, применимость к гражданам РФ, даты проверки и спорные места.
5. После проверки расширить каталог до 34 стран из `docs/research/russian-relocation-destinations-2026-06-18.md`.
6. Непроверенные страны и маршруты оставлять в статусах `reference_only`, `needs_review` или `stale`, а не выдавать как проверенные.

## Критерий готовности системы

Система считается управляемой, когда:

- новая страна добавляется без изменения React-компонентов;
- новый маршрут добавляется через данные и правила;
- каждое утверждение имеет источник;
- можно отличить проверенное от черновика;
- есть автоматическая проверка структуры данных;
- результаты показывают не только "подходит", но и "что нужно изменить";
- агент-сборщик может работать независимо от разработчика.
