# AGENTS.md

## Язык и стиль работы

- Отвечай пользователю на русском языке, если он явно не попросил иначе.
- Не используй лишние англицизмы в объяснениях для пользователя. Названия библиотек, сервисов и официальных технологий можно оставлять как есть.
- Работай из корня проекта. Не создавай файлы в родительских директориях.
- Сначала собирай контекст, затем меняй код.
- Не меняй незапрошенные файлы и не выполняй разрушительные команды без явного запроса.
- Не печатай секреты, токены, пароли, cookies, приватные ключи и содержимое `.env`.

## Назначение продукта

Проект строит сайт, а затем приложение для подбора маршрутов переезда из РФ в конкретные страны.

Главная логика:

1. Пользователь проходит анкету.
2. Система определяет цели, ограничения и исходные данные.
3. Система подбирает страны и маршруты.
4. Для каждой связки `пользователь + цель + страна + маршрут` считается сложность.
5. Пользователь получает пошаговый план, список документов, сроки, риски, официальные источники и возможность обратиться к проверенному специалисту.

Страна не является первым фильтром. Сначала идет анкета пользователя, затем подбор стран.

## Ключевое ограничение по ИИ

- Не делай постоянное использование ИИ обязательным для основного пользовательского пути.
- Подбор стран, расчет сложности, маршруты, документы и предупреждения должны работать на структурированных правилах, базе данных и проверяемых источниках.
- ИИ можно использовать только как необязательный внутренний инструмент для черновиков описаний, нормализации текста источников или помощи администраторам.
- Нельзя использовать ИИ как единственный источник правовой, визовой, налоговой или миграционной рекомендации.
- Любой пользовательский вывод по правилам переезда должен иметь дату обновления и ссылку на источник.

## Юридическая аккуратность

- Продукт не должен выдавать себя за юридическую консультацию.
- На страницах маршрутов и перед обращением к специалисту показывай короткое предупреждение: информация справочная, правила меняются, окончательное решение принимает компетентный орган.
- Разделяй:
  - официальные источники;
  - редакционные пояснения проекта;
  - мнение или консультацию партнера.
- Для каждой страны и маршрута храни:
  - источник;
  - дату последней проверки;
  - применимость к гражданам РФ;
  - условия;
  - исключения;
  - уровень уверенности данных.

## Рекомендуемый стек

По умолчанию используй:

- Next.js с App Router;
- TypeScript;
- Supabase Postgres для данных;
- Supabase Auth для регистрации, входа и восстановления доступа;
- Supabase Row Level Security для ограничения доступа к данным пользователя;
- React Hook Form для сложных форм;
- Zod для проверки входных данных;
- shadcn/ui, Radix UI, Tailwind CSS и Lucide React для интерфейса;
- Playwright для проверок пользовательских сценариев;
- Vitest или Jest для модульных тестов правил подбора;
- Vercel для сайта, если пользователь не выберет другой хостинг.

Не пиши собственную авторизацию, хранение паролей, выпуск токенов и восстановление доступа. Если нет особой причины, используй Supabase Auth.

## Архитектурные правила

- Разделяй справочные данные, правила подбора, пользовательскую анкету и интерфейс.
- Не смешивай визовую логику с компонентами интерфейса.
- Правила сложности должны быть тестируемыми чистыми функциями.
- Контент стран и маршрутов должен быть редактируемым через административную часть или структурированные файлы до появления полноценной админки.
- Не хардкодь страны и маршруты внутри компонентов страниц.
- Не добавляй новые зависимости без необходимости.
- Если библиотека уже решает задачу стандартно, используй ее вместо самописного решения.

## Данные стран и маршрутов

- Не дополняй страны, маршруты, законы, визовые правила и миграционные графы хаотичным ручным поиском прямо во время разработки интерфейса.
- Для существенного пополнения данных используй навык `deepseek-opencode`: DeepSeek через opencode готовит черновик, агент проверяет полноту, источники и структуру.
- Для каждого факта о стране или маршруте требуй официальный источник, дату проверки, применимость к гражданам РФ, уверенность, исключения и нерешенные вопросы.
- DeepSeek не является источником истины. Юридические, визовые, налоговые и миграционные утверждения должны быть сверены с первичными источниками перед публикацией.

## Деплой

- Базовая модель деплоя: сначала изменения делаются и проверяются локально у пользователя.
- После проверок: `git add`, осмысленный коммит, `git push`.
- На сервере изменения подтягиваются через `git pull` из репозитория, затем выполняется сборка и перезапуск нужных контейнеров или служб.
- Не правь код напрямую на сервере, кроме аварийной диагностики. Если аварийная правка все же нужна, перенеси ее обратно в репозиторий отдельным коммитом.
- После серверного обновления проверяй статус служб, логи и доступность HTTP/HTTPS-страниц.

## Дизайн и пользовательский опыт

- Продукт должен ощущаться как надежный навигатор по документам, а не как рекламный лендинг.
- Первая страница должна сразу вести к действию: начать анкету, посмотреть направления, продолжить сохраненный маршрут.
- Не делай декоративный лендинг как основной экран.
- Используй спокойный, служебный, доверительный стиль.
- Формы должны быть понятными, короткими и пошаговыми.
- Для сложной анкеты используй один главный вопрос на экран или небольшие логические группы.
- Всегда показывай прогресс, возможность вернуться назад и сохранение черновика.
- На мобильных экранах не должно быть горизонтальной прокрутки.
- Все поля формы должны иметь подписи, ошибки рядом с полем и доступную навигацию с клавиатуры.
- Не используй эмодзи как иконки. Используй Lucide React.
- Не используй крупные декоративные градиенты, размытые пятна, карточки внутри карточек и рекламную стилистику.

## Проверки перед финалом

Перед финальным ответом после изменений запусти релевантные проверки:

- форматирование;
- проверку типов;
- линтер;
- модульные тесты правил подбора;
- сборку;
- пользовательские сценарии через Playwright, когда интерфейс уже существует.

Если проверку нельзя запустить, явно скажи почему.
