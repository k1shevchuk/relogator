# Relocation Navigator Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` when implementing this plan task by task. Steps use checkbox syntax for tracking.

**Цель:** построить сайт, а затем приложение, которое помогает гражданам РФ подобрать реалистичные маршруты переезда по личной анкете, сложности цели и проверенным официальным источникам.

**Архитектура:** первая версия должна быть веб-сервисом с анкетой, расчетом подходящих стран, базой маршрутов, личным кабинетом и партнерскими обращениями к специалистам. Основная логика подбора должна быть основана на структурированных правилах и данных, а не на постоянном использовании ИИ.

**Стек:** Next.js, TypeScript, Supabase Postgres, Supabase Auth, Supabase Row Level Security, React Hook Form, Zod, shadcn/ui, Radix UI, Tailwind CSS, Lucide React, Playwright, Vitest или Jest.

**Деплой:** изменения делаются локально, затем `git add`, коммит, `git push`; сервер обновляется через `git pull`, сборку и перезапуск служб или контейнеров. Прямые правки кода на сервере запрещены, кроме аварийной диагностики с обязательным переносом обратно в репозиторий.

**Пополнение стран и маршрутов:** новые данные по странам, законам, въезду, выезду, ВНЖ и маршрутам готовятся через навык `deepseek-opencode`: DeepSeek через opencode делает черновик с источниками, агент проверяет структуру, первичные источники, применимость к гражданам РФ и уровень уверенности. DeepSeek не является источником истины.

## 1. Продуктовая рамка

### 1.1. Рабочее название

`Relocation Navigator` для внутреннего использования. Перед публичным запуском название можно заменить.

### 1.2. Основная идея

Пользователь не выбирает страну сразу. Он сначала отвечает на вопросы о себе, целях и ограничениях. Сервис показывает подходящие страны и маршруты, затем объясняет сложность каждого варианта.

Пример результата:

| Страна  | Маршрут                                | Сложность | Почему подходит                | Главный риск                          |
| ------- | -------------------------------------- | --------- | ------------------------------ | ------------------------------------- |
| Армения | въезд и проживание до 180 дней         | простая   | не нужна виза, близкая среда   | ограничение срока пребывания          |
| Сербия  | ВНЖ через работу или бизнес            | средняя   | понятная подача, есть партнеры | нужен комплект документов             |
| Таиланд | долгосрочная виза для удаленной работы | сложная   | есть подходящий тип визы       | нужны деньги на счете и подтверждения |

### 1.3. Для кого первая версия

- граждане РФ;
- люди, которые хотят уехать в ближайшие 1-12 месяцев;
- удаленные специалисты;
- семьи с детьми;
- люди с ИП, ООО, трудовым договором или подтверждаемым доходом;
- люди, которым нужна не статья, а личный маршрут и список действий.

### 1.4. Не целевая аудитория первой версии

- люди, которые уже наняли юриста и ждут только подачи;
- беженские и убежищные кейсы;
- сложные уголовные, санкционные и политические случаи;
- крупные корпоративные релокации;
- полноценное сопровождение “под ключ”.

### 1.5. Главная ценность

- меньше хаоса и противоречивой информации;
- понятный путь “что делать сейчас”;
- сравнение стран под личную цель;
- ссылки на официальные источники;
- предупреждения о частых ошибках;
- подключение специалиста только на тех шагах, где это действительно нужно.

## 2. Цели переезда и логика сложности

### 2.1. Основные цели пользователя

Первая версия должна поддерживать такие цели:

1. Быстро уехать на 1-3 месяца.
2. Легально жить 3-12 месяцев.
3. Получить временное проживание или ВНЖ.
4. Уехать с удаленной работой.
5. Уехать через работу в местной компании.
6. Открыть ИП, компанию или малый бизнес.
7. Переехать с семьей.
8. Переехать с животными.

Цели “получить гражданство” и “переехать в страну с тяжелой визовой системой” добавить как справочные разделы, но не делать главной обещанной ценностью первой версии.

### 2.2. Принцип оценки

Сложность считается не для страны вообще, а для связки:

`пользователь + цель + страна + маршрут`

Одна и та же страна может быть простой для короткого проживания и сложной для ВНЖ.

### 2.3. Шкала сложности

| Уровень | Название      | Значение                                                          |
| ------- | ------------- | ----------------------------------------------------------------- |
| 1       | очень простой | мало документов, быстрый въезд, низкий риск отказа                |
| 2       | простой       | нужны базовые документы, правила понятные                         |
| 3       | средний       | нужны деньги, договор, подтверждение дохода или специалист        |
| 4       | сложный       | много документов, высокая цена ошибки, заметный риск отказа       |
| 5       | очень сложный | нужен работодатель, баллы, квоты, крупные деньги или долгие сроки |

### 2.4. Факторы сложности

Каждый маршрут должен оцениваться по факторам:

- въезд;
- нужный срок пребывания;
- основание для пребывания;
- минимальный доход;
- накопления;
- пошлины и расходы;
- количество документов;
- переводы, апостили, нотариальные действия;
- сроки подготовки;
- вероятность отказа;
- возможность продления;
- семья;
- дети;
- животные;
- банковский счет;
- регистрация адреса;
- местные налоги;
- возможность работы или бизнеса;
- наличие официального источника;
- дата обновления данных.

### 2.5. Пример правила оценки

Если пользователь хочет жить 6-12 месяцев, но у него нет подтверждаемого дохода, нет работодателя и маленький бюджет, маршруты через визы удаленной работы должны получать высокий уровень сложности или пометку “не подходит сейчас”.

Если пользователь хочет уехать на 1-3 месяца, имеет действующий загранпаспорт и выбирает страну с безвизовым въездом для граждан РФ, маршрут может быть простым даже без ВНЖ.

### 3.2. Средние и тяжелые направления для сравнения

Эти страны нужны, чтобы пользователь понимал, почему маршрут сложнее, но их не нужно глубоко реализовывать в первой версии.

Тяжелые:

- США;

- Великобритания;

У каждой страны должен быть статус:

- `draft`: черновик, не показывать в публичном подборе;
- `reviewed`: проверено редактором;
- `partner_reviewed`: проверено партнером-специалистом;
- `stale`: устарело, показывать предупреждение;
- `archived`: скрыто из подбора.

## 4. Анкета пользователя

### 4.1. Общие правила

- Анкета должна быть пошаговой.
- Не спрашивать лишнего до того, как это влияет на подбор.
- Сохранять черновик после каждого шага.
- Показывать прогресс.
- Давать возможность вернуться назад.
- На сложных вопросах объяснять, зачем вопрос нужен.

### 4.2. Первый блок вопросов

1. Какую цель вы хотите решить?
   - уехать быстро;
   - пожить несколько месяцев;
   - получить ВНЖ;
   - работать;
   - открыть бизнес;
   - переехать с семьей;
   - пока сравниваю варианты.

2. Когда нужно уехать?
   - в течение 2 недель;
   - в течение 1 месяца;
   - в течение 3 месяцев;
   - в течение года;
   - пока без срока.

3. На какой срок хотите уехать?
   - до 1 месяца;
   - 1-3 месяца;
   - 3-6 месяцев;
   - 6-12 месяцев;
   - больше года;
   - хочу путь к постоянному статусу.

4. Есть ли действующий загранпаспорт РФ?
   - да, срок больше 18 месяцев;
   - да, срок 6-18 месяцев;
   - да, срок меньше 6 месяцев;
   - нет.

5. Едете один или с кем-то?
   - один;
   - с партнером;
   - с детьми;
   - с родителями;
   - с животными.

### 4.3. Финансовый блок

1. Есть ли подтверждаемый регулярный доход?
2. Какой примерный месячный доход можно подтвердить документами?
3. Есть ли накопления?
4. Есть ли трудовой договор?
5. Есть ли ИП или ООО?
6. Готов ли пользователь открыть компанию за рубежом?

### 4.4. Документальный блок

1. Есть ли свидетельства о браке и рождении детей?
2. Готов ли пользователь делать переводы и апостили?
3. Есть ли справка о несудимости?
4. Есть ли дипломы или документы о квалификации?
5. Есть ли медицинские документы, если они важны для маршрута?

### 4.5. Бытовой блок

1. Нужна ли школа или детский сад?
2. Нужен ли банковский счет?
3. Нужна ли регистрация бизнеса?
4. Важна ли русскоязычная среда?
5. Важна ли низкая стоимость жизни?
6. Важен ли теплый климат?

## 5. Результаты подбора

### 5.1. Экран результатов

Показывать не просто страны, а маршруты.

Обязательные поля карточки:

- страна;
- маршрут;
- уровень сложности;
- почему подходит;
- почему может не подойти;
- примерные сроки;
- примерные расходы;
- список ключевых документов;
- дата последней проверки;
- источники;
- действие “открыть пошаговый план”;
- действие “задать вопрос специалисту”.

### 5.2. Фильтры результатов

- проще всего;
- быстрее всего;
- дешевле всего;
- подходит для семьи;
- подходит для удаленной работы;
- подходит для бизнеса;
- меньше документов;
- есть партнерская помощь.

### 5.3. Пошаговый план маршрута

Каждый маршрут должен иметь шаги:

1. Что проверить до решения.
2. Что подготовить в РФ.
3. Что сделать перед выездом.
4. Что сделать при въезде.
5. Что сделать в первые 7 дней.
6. Что сделать в первые 30 дней.
7. Что сделать до окончания разрешенного срока.
8. Как продлить или перейти на другой статус.

У каждого шага:

- описание действия;
- список документов;
- официальные источники;
- частые ошибки;
- когда нужен специалист;
- связанные партнерские услуги.

## 6. Партнерская модель

### 6.1. Роли партнеров

Партнеры могут быть:

- миграционные консультанты;
- юристы;
- налоговые консультанты;
- переводчики;
- специалисты по апостилю;
- компании по открытию бизнеса;
- страховые посредники;
- помощники по жилью;
- специалисты по перевозке животных;
- консультанты по банкам и счетам.

### 6.2. Как показывать партнеров

Не делать главную страницу витриной агентств. Партнер должен появляться в контексте конкретного шага.

Примеры:

- “Проверить комплект документов для ВНЖ в Сербии”.
- “Подобрать страховку для подачи”.
- “Проконсультироваться по налоговому резидентству”.
- “Помощь с переводом и заверением документов”.

### 6.3. Профиль партнера

Поля:

- название;
- страна;
- города;
- услуги;
- языки;
- цена от;
- средний срок ответа;
- подтвержденные документы или лицензии;
- рейтинг;
- количество завершенных обращений;
- правила возврата;
- способ связи;
- статус проверки.

### 6.4. Заявка партнеру

Пользователь может отправить партнеру:

- выбранный маршрут;
- ответы анкеты;
- список уже готовых документов;
- вопрос;
- контакт для связи.

Нельзя отправлять данные партнеру без явного согласия пользователя.

### 6.5. Монетизация партнеров

Разрешенные модели:

- комиссия за обращение;
- комиссия за успешную услугу;
- платное размещение после ручной проверки;
- платная проверка документов;
- платная консультация.

Не показывать партнеров как “официально рекомендованных”, если нет строгой процедуры проверки.

## 9. Правила подбора

### 9.1. Базовый алгоритм

1. Получить анкету пользователя.
2. Определить цель и ограничения.
3. Отфильтровать маршруты, которые явно не подходят.
4. Для каждого маршрута посчитать сложность.
5. Объяснить причины оценки.
6. Отсортировать маршруты.
7. Показать пользователю лучшие варианты и причины.

### 9.2. Признаки неподходящего маршрута

Маршрут не подходит, если:

- срок паспорта меньше минимального требования;
- цель пользователя требует ВНЖ, а маршрут покрывает только короткий въезд;
- нужен подтверждаемый доход, а пользователь указал, что подтвердить доход не может;
- нужен работодатель, а пользователь ищет самостоятельный переезд;
- семья или дети не поддерживаются маршрутом;
- данные маршрута устарели и не прошли проверку.

### 9.3. Объяснимость

Каждый результат должен объяснять:

- какие ответы анкеты повлияли на подбор;
- какие требования маршрута пользователь уже выполняет;
- какие требования остаются;
- что делает маршрут сложнее;
- какие документы нужно подготовить первыми.

### 9.4. Тестируемость

Правила подбора должны покрываться модульными тестами:

- короткий переезд без ВНЖ;
- удаленная работа с подтверждаемым доходом;
- семья с детьми;
- паспорт истекает через 5 месяцев;
- отсутствие подтверждаемого дохода;
- срочный выезд за 2 недели;
- переезд с животным;
- бизнес-маршрут.

### 10.3. Результаты

Элементы:

- список маршрутов;
- сортировка;
- фильтры;
- сравнение;
- объяснение сложности;
- кнопка сохранения;
- переход в пошаговый план.

### 10.4. Страница маршрута

Элементы:

- краткий итог;
- уровень сложности;
- применимость;
- список документов;
- пошаговый план;
- официальные источники;
- частые ошибки;
- партнерская помощь;
- дата обновления.

### 10.5. Личный кабинет

Элементы:

- анкета;
- сохраненные маршруты;
- статус подготовки;
- документы, которые нужно собрать;
- уведомления;
- обращения к партнерам.

### 10.6. Партнерский раздел

Элементы:

- список обращений;
- профиль партнера;
- услуги;
- статусы заявок;
- история ответов.

### 10.7. Административный раздел

Элементы:

- страны;
- маршруты;
- шаги;
- источники;
- правила сложности;
- партнеры;
- проверки контента;
- пометка устаревших данных.

Административный раздел можно сделать простым, но он должен быть защищен ролями.

## 11. Авторизация и доступ

### 11.1. Решение по умолчанию

Использовать Supabase Auth.

Причины:

- не нужно писать собственное хранение паролей;
- есть готовые сценарии входа и восстановления доступа;
- удобно связать пользователя с Postgres;
- можно защищать строки через Row Level Security;
- подходит для сайта и будущего приложения.

### 11.2. Роли

- `guest`: не вошел, может пройти короткую анкету и увидеть ограниченный результат.
- `user`: вошел, может сохранять анкету, маршруты и обращения.
- `partner`: видит только свои заявки и услуги.
- `editor`: редактирует страны, маршруты, источники.
- `admin`: управляет ролями, партнерами и публикацией.

### 11.3. Обязательные правила доступа

- Пользователь видит только свои анкеты, сохраненные маршруты и обращения.
- Партнер видит только заявки, которые пользователь отправил этому партнеру.
- Редактор не должен видеть личные данные пользователей без необходимости.
- Администраторские действия логируются.
- Секретные ключи Supabase не попадают в клиентский код.

## 12. Платежи

### 12.1. Первая версия

Не усложнять платежи до проверки спроса. Разрешить ручную продажу платных маршрутов или консультаций.

### 12.2. Будущая версия

Платежный слой должен быть отделен интерфейсом:

- `createCheckout`;
- `getPaymentStatus`;
- `grantRouteAccess`;
- `refundPayment`.

Провайдера выбрать отдельно под юридическое лицо и страну приема платежей. Не привязывать бизнес-логику к одному провайдеру.

## 13. Дизайн-система

### 13.1. Визуальный характер

Сервис должен выглядеть:

- спокойным;
- надежным;
- понятным;
- аккуратным;
- не рекламным;
- не “криптовалютным”;
- не развлекательным.

### 13.2. Цвета

Рекомендуемые базовые цвета:

- фон: `#F8FAFC`;
- поверхность: `#FFFFFF`;
- основной текст: `#0F172A`;
- вторичный текст: `#475569`;
- основной акцент: `#2563EB`;
- успешное состояние: `#15803D`;
- предупреждение: `#B45309`;
- ошибка: `#B91C1C`;
- граница: `#E2E8F0`.

Не строить весь интерфейс вокруг одного синего цвета. Использовать цвет только для смысла: действие, успех, предупреждение, ошибка, нейтральный статус.

### 13.3. Типографика

- Использовать современный гротеск: Inter, Geist или системный шрифт.
- Не использовать декоративные шрифты.
- Минимальный размер основного текста на мобильных экранах: 16px.
- Длина строки в текстовых блоках: 65-75 символов.
- Не масштабировать размер шрифта от ширины экрана.
- Межбуквенное расстояние не делать отрицательным.

### 13.4. Компоненты

Использовать готовые компоненты:

- кнопки;
- поля ввода;
- переключатели;
- радиокнопки;
- чекбоксы;
- вкладки;
- таблицы;
- раскрывающиеся блоки;
- диалоги;
- уведомления;
- индикатор прогресса;
- карточки маршрутов.

Не рисовать базовые элементы вручную, если они есть в shadcn/ui или Radix UI.

### 13.5. Формы

Следовать подходам GOV.UK Design System и Nielsen Norman Group:

- один вопрос на страницу, если вопрос сложный;
- логические группы для простых связанных вопросов;
- поля в одну колонку;
- подпись рядом с каждым полем;
- ошибки рядом с проблемным полем;
- видимое состояние фокуса;
- короткие объяснения рядом со сложными вопросами;
- перед длинной анкетой показать, какие документы могут понадобиться.

## 14. Инструкция для ИИ-дизайнера

Если Codex, Figma Make, v0 или другой генератор интерфейса используется для создания макетов, ему нужно давать полный контекст. Плохая инструкция “сделай красивый сайт для переезда” приведет к декоративному лендингу. Хорошая инструкция должна содержать продуктовую задачу, аудиторию, ограничения, список экранов, структуру данных, стиль и запреты.

### 14.1. Что обязательно передавать

1. Кто пользователь.
2. Какую задачу он решает.
3. Какой экран нужно сделать.
4. Какие данные есть на экране.
5. Какие действия доступны.
6. Какие состояния нужны: пусто, загрузка, ошибка, частичный результат, устаревшие данные.
7. Какой стиль нужен.
8. Какие компоненты использовать.
9. Чего нельзя делать.
10. Как проверить результат.

### 14.2. Готовая инструкция для генерации дизайна

```text
Создай интерфейс веб-сервиса для подбора маршрутов переезда из РФ.

Продукт: надежный навигатор по переезду, документам, срокам и маршрутам. Это не рекламный лендинг и не юридическая фирма. Главная ценность: пользователь проходит анкету, получает подходящие страны и пошаговые маршруты с официальными источниками.

Аудитория: граждане РФ, которые хотят уехать на 1-12 месяцев, получить временное проживание или подготовить переезд с семьей. Пользователь тревожится, боится ошибок в документах и хочет понятный план.

Стиль: спокойный, служебный, доверительный, современный. Светлый интерфейс, много воздуха, четкая иерархия, минимум декоративности. Цвета: фон #F8FAFC, поверхность #FFFFFF, текст #0F172A, вторичный текст #475569, акцент #2563EB, успех #15803D, предупреждение #B45309, ошибка #B91C1C, граница #E2E8F0.

Компоненты: использовать shadcn/ui, Radix UI, Tailwind CSS, Lucide React. Не использовать эмодзи как иконки. Не использовать крупные градиенты, декоративные пятна, карточки внутри карточек и маркетинговые иллюстрации.

Экран: результаты подбора после анкеты.

Данные на экране:
- цель пользователя: легально жить 3-12 месяцев;
- паспорт действует 14 месяцев;
- есть удаленный доход;
- едет с партнером;
- бюджет средний;
- нужны страны с понятным продлением.

Покажи:
- краткое резюме анкеты;
- 5 карточек маршрутов;
- для каждой карточки страну, маршрут, сложность, сроки, расходы, почему подходит, что мешает, дату обновления;
- фильтры: проще всего, быстрее всего, дешевле всего, подходит для семьи, меньше документов;
- кнопку “Открыть пошаговый план”;
- кнопку “Задать вопрос специалисту”;
- предупреждение, что информация справочная и правила меняются.

Состояния:
- загрузка;
- нет подходящих маршрутов;
- данные устарели;
- ошибка сохранения;
- пользователь не вошел и может сохранить результат после входа.

Требования:
- мобильная версия в первую очередь;
- поля и кнопки не должны прыгать при наведении;
- все кнопки имеют видимый фокус;
- текст не выходит за контейнеры;
- основной сценарий должен быть понятен без поясняющего текста о возможностях продукта.
```

## 18. Что не делать

- Не начинать с мобильного приложения до проверки веб-версии.
- Не делать “каталог стран” вместо персонального подбора.
- Не писать собственную авторизацию.
- Не хранить пароли самостоятельно.
- Не использовать ИИ как обязательный механизм принятия решений.
- Не давать юридических обещаний.
- Не показывать партнеров как официальную гарантию результата.
- Не добавлять 30 стран в первую версию.
- Не делать интерфейс похожим на рекламный лендинг.
- Не скрывать дату обновления источников.
